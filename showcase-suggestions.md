# Showcase Suggestions for Paper Trading Bot

Generated by /showcase on 2026-04-24. Quick wins to make this project look stronger to recruiters and hiring managers.

## Critical (do these before sharing)

- [ ] **Add a dashboard screenshot or GIF to the README hero.** The Streamlit dashboard is the most visual part of the project — without a screenshot, recruiters scroll right past. Run the dashboard, capture a 5-10 second GIF (LICEcap or ScreenToGif), drop it in `docs/demo.gif`, and replace the `<!-- DEMO -->` placeholder.
- [ ] **Run `python run_backtest.py` and paste real numbers into the "Backtest Results" section.** Right now the README claims 5 strategies + ML ensemble, but a recruiter sees no evidence any of them work. Three rows of real numbers (one per symbol) does more for credibility than the strategy table.
- [ ] **Fill in "Why I built this / What I learned"** (the `<!-- TODO -->` section in `README.draft.md`). Two or three concrete sentences in your voice. This is the section that distinguishes a portfolio project from a tutorial walkthrough.
- [ ] **Decide what to do with `CLAUDE.md` in the repo root.** It's currently a generic Claude prompt file with no project-specific content. A recruiter glancing at the repo root sees it and wonders "did they hand-write this code?" Either move it to `.claude/CLAUDE.md` (so it's hidden from the default GitHub view) or delete it.

## Nice to have (1-2 hours each)

- [ ] **Add badges to the README hero** (under the title, above the tagline): build status (link to your existing `.github/workflows/ci.yml`), license (MIT), Python version (3.11+). One row of badges signals "this is a real project, not abandoned."
- [ ] **Add a coverage badge.** `pytest-cov` is already in dev dependencies — wire it into CI, push results to Codecov or Coveralls (both free for public repos), drop the badge under the build status one.
- [ ] **Pin a "starter" strategy in `.env.example`** with comments explaining what each strategy does. Right now someone cloning the repo doesn't know which `STRATEGY=` value to start with.
- [ ] **Add a terminal GIF showing a backtest run** alongside the dashboard one — five strategies' output scrolling past with concrete numbers is its own kind of demo. Two assets are stronger than one.

## Project polish (ongoing)

- [ ] **Increase test coverage on the strategies module.** You have `test_strategies.py` but with 5 strategies + a base class, one test file likely under-tests them. A test file per strategy (or parametrized tests) would let you claim coverage numbers in the README.
- [ ] **Add docstrings to public methods on `BacktestEngine`, `OrderManager`, and each strategy's `generate_signal`.** Short ones. They make the code feel professional when a recruiter clicks through src/.
- [ ] **Consider a `notebooks/` README.** You have a `notebooks/` directory — if it has anything in it, a one-line index file ("01-strategy-comparison.ipynb: backtests all strategies on AAPL/MSFT/TSLA, plots equity curves") makes the directory worth clicking into.
- [ ] **Tag a release.** A `v0.1.0` git tag + GitHub release page with a short changelog signals maturity. Even one release looks better than none.
